# 数据类型

Java中一共有两种数据类型：原始数据类型、引用数据类型。

# 原始数据类型

Java是完全面向对象的，但原始类型不是对象，其具有显式范围和数学行为。

byte、short、int、long都是有符号二进制补码整数，有范围限制。

float、double是IEEE 754 浮点数，不能用于精确计算。

类型 大小 默认值
byte 1Byte，8bit，[-128,127] 0
short 2Byte，[-32768,32767] 0
int 4Byte，[-2^31,2^31-1]
[-2,147,483,648~2,147,483,647]
相当于+-21亿多，无符号则翻倍有40亿		 0

boolean的大小问题：《Java虚拟机规范》给出了参考，有可能是1Byte，或4Byte，关键还要看具体JVM的实现。书中说明，boolean编译后会变为int类型，因此占4字节，而boolean数组编译后是byte数组，因此一个boolean站1字节。占4字节是因为当下CPU一次处理数据以及CPU寻址都是32位，因此4字节具有高效存取的特点。

# Java的参数传递是值传递还是引用传递？

Java参数传递方式本质只有值传递。

  1. 如果参数是基本类型，传参会拷贝值

  2. 如果参数是引用类型，传参会拷贝引用地址，不会动堆中的实例对象

共性：都是值传递，但传递的东西不一样。

从下面的案例可以看明白：

public static void main(String[] args) {
    StringBuilder sb = new StringBuilder("iphone");
    foo(sb);
    System.out.println(sb); // iphone

    foo2(sb);
    System.out.println(sb); // iphone13

    int n = 0;
    foo(n);
    System.out.println(n); // 0
}

private static void foo(StringBuilder sb) {
    sb = new StringBuilder("ipad");
    System.out.println(sb); // ipad
}

private static void foo2(StringBuilder sb) {
    sb.append("13");
    System.out.println(sb); // iphone13
}

private static void foo(int n) {
    n = 1;
    System.out.println(n); // 1
}

# 整型之间的比较（包括Integer和int）

  1. Integer类型注意不要使用==比较，而应该是用equals

    ==都比较的是变量的值，由于Integer是对象类型，其值为堆地址。 但由于Integer的享元模式设计，值在[-128,127]区间内都会返回同一个Integer对象，因此区间内的对象使用==比较结果仍然正确，容易造成误解。

  2. Integer和int比较会拆箱

    包装类型和原始类型使用==比较也能获得正确的结果，是因为包装类型会拆箱成普通类型再进行比较。虽然比较结果正确，但容易产生bug，仍要避免。

# long和Long

Long的包装类型通过LongCache默认缓存了[-128,127]区间内的值（与Integer一样），因此拥有一样类似的特性。

# Switch-case条件控制语句

开发中偶尔会使用到Switch，但通常大家只了解以下常规知识点：

  • switch的case中一定要记得使用break
  • 如果case中没有break语句，会继续往下执行其他case、default语句。

这里列举两个小坑，来帮你深入熟悉Switch：

default可以写在任何位置，不一定作为switch的最后条件，但这种约束常常出现在编码规约中，用于避免大家踩坑

int i = 0;
switch (i) {
    default:
        System.out.println("default");
    case 0:
        System.out.println("0");
    case 1:
        System.out.println("1");
}
// 结果输出：0 1

如果匹配到了对应的case后，没有加break语句，会继续往下执行，default条件不在下面，则不会执行，注意是继续往下执行！

若没有匹配到case条件，则会走default条件分支，若没有break语句，仍然会继续往下执行，而不会结束！

int i = 0;
switch (i) {
    default:
        System.out.println("default");
    case 2:
        System.out.println("2");
    case 3:
        System.out.println("3");
}
// 结果输出：default 2 3

# 引用（四种引用类型）

强引用/弱引用/软引用/幻想引用

# 参考地址

Java四种引用包括强引用，软引用，弱引用，虚引用 (opens new window)

# 强引用

只要引用存在, 垃圾回收器永远不会回收 这也是我们常用的编码方式

Object obj = new Object();

# 软引用

非必须引用, 内存溢出之前进行回收

Object obj = new Object();
SoftReference<Object> sf = new SoftReference<Object>(Object);
obj = null;
sf.get();//有时会返回null

sf是对obj的一个软引用, 通过sf.get()可以获取这个对象, 这个对象被标记为需要回收的对象时, 则返回null. 用途: 软引用主要用来实现类似缓存的功能, 在内容足够的情况下, 直接通过软引用取值, 无需从繁忙的真是数据来源查询, 提升速度; 当内存不足, 会自动删除这部分数据, 从真实来源查询.

# 弱引用

第二次垃圾回收时回收

Object obj = new Object();
WeakReference<Object> wf = new WeakReference<Object>(obj);
obj = null;
wf.get();//有时会返回null
wf.isEnQueued();//返回是否被垃圾回收器标记为即将回收的垃圾

用途: 主要用于监控对象是否已经被垃圾回收器标记为即将回收的垃圾, 通过isEnQueued()

# 虚引用

无法通过引用取到对象值, 垃圾回收时回收
虚引用每次在垃圾回收时都会被回收, 通过虚引用的get永远无法获取到数据, 因此也被成为幽灵引用.

Object obj = new Object();
PhantomReference<Object> pf = new PhantomReference<Object>(obj);
obj = null;
pf.get();//永远返回null
pf.isEnQueued();//返回是否从内存中已经删除

用途: 主要用于检测对象是否已经从内存中删除.

# Java的类加载机制

概念：

Java虚拟机把描述类的数据从class文件读取到内存，并对数据进行校验、转换、解析、初始化，最终成为虚拟机可以直接使用的Java类型，整个过程被称为虚拟机的类加载机制。

Java类加载机制的优势：

静态语言需要在编译时做好类型连接，而Java是在运行时做类型的加载、连接、初始化。虽然这种策略会让提前编译产生额外的困难（具体指什么？），且让运行时类加载带来了额外的性能开销，但缺提供了极高的扩展性和灵活性。

动态扩展的特性就是由此策略的类加载机制提供的，例如：

  1. 面向接口的代码，可以在运行期再指定其实际的实现类（可以从外部网络加载的实现类）
  2. 原始的Applet、JSP到复杂的OSGI都依赖运行时类加载机制

# 类型生命周期的顺序

五个阶段是确定的：加载、验证、准备、初始化、卸载。

但解析阶段顺序不是固定的，有可能发生在初始化之后，是为了支持Java语言的运行时绑定特性。

什么情况开始类加载的第一阶段--加载：由JVM自己实现，《Java虚拟机规范》并没有规定。

但《Java虚拟机规范》对初始化阶段做了强制规定，有6种情况必须对类进行初始化（那加载、验证、准备必然要在这之前），统称为--主动引用，场景如下：

  1. 遇到new、getstatic、putstatic、invokestatic四条字节码指令时，如果类型没有初始化过，先触发其初始化阶段。生成这四条指令的典型场景有：
    1. new 实例化对象时
    2. 读取或设置一个类型的静态字段（被final修饰、在编译期把结果放入到常量池的静态字段除外）
    3. 调用一个类型的静态方法时
  2. 使用reflect包的方法对类型进行反射调用时
  3. 初始化类时，其父类还未初始化，优先初始化父类
  4. JVM启动时必须指定执行主类（main方法类），JVM优先初始化它
  5. 使用JDK7加入的动态语言支持的java.lang.invoke.MethodHandle时，其实例解析的结果为REF_getStatic、REF_putStatic、REF_invokeStatic、REF_newInvokeSpecial四种类型的方法句柄，并且句柄对应的类没有进行过初始化时，需先触发其初始化
  6. 接口定义了default方法时，其实现类发生初始化时，其接口要优先初始化

以上是主动引用的方式触发类的初始化阶段，除此之外，所有引用类型的方式都不会触发初始化阶段，故被称为被动引用。例如：

  1. 通过子类引用父类的静态字段，不会导致子类被初始化。

    使用静态字段，只有直接定义这个字段的类才会被初始化，因此不会触发子类的初始化。至于是否会触发子类的加载、验证阶段，《规范》中未规定，HotSpot通过设置-XX: +TraceClassLoading可观察到此操作会导致子类的加载。

  2. 通过定义数组类引用类，不会触发此类型的初始化

    但会触发名为"[Lxxx.package.name.Class"类的初始化阶段，它是一个由JVM自动生成，直接继承于java.lang.Object的子类，创建动作由字节码指令newarray触发。数组类的属性和方法都实现在这个类中（例如length属性和clone()方法），Java对操作数组相比于其他语言更安全，很大程度也是因为这个类包装了数组元素的访问。

    越界检查不是封装在数组类中，而是封装在数组访问的xaload、xastore字节码指令中。

  3. 常量在编译阶段会存入调用类的常量池中（通过常量传播优化），本质上没有引用到定义常量的类，故不会触发它的初始化

# 接口有没有类的初始化过程？

有，与类是一致的。

接口与类初始化阶段存在区别的场景是（前面6种场景的第三种）：接口初始化时，不要求其父接口全部都初始化完成，只有真正使用父接口时（如引用接口中定义的常量）才会初始化。

注意：接口中不能使用static{}语句块，但编译器仍然会生成类构造器clinit()用于初始化接口中定义的成员变量。

# 类、方法、代码块、属性的加载顺序

  1. 加载父类的静态字段、静态语句块
  2. 子类的静态字段、静态语句块
  3. 父类的普通变量、语句块
  4. 父类的构造方法
  5. 子类的普通变量、语句块
  6. 子类的构造方法

注意：

  1. 静态语句块和静态变量被初始化的顺序与代码先后顺序有关
  2. 静态语句块、静态变量，只会被加载一次（类初始化时）
  3. 构造方法只在创建实例时加载一次

# 使用contains方法去重

private static void removeDuplicate(List<String> list) {
    List<String> result = new ArrayList<String>(list.size());
    for (String str : list) {
        if (!result.contains(str)) {
            result.add(str);
        }
    }
    list.clear();
    list.addAll(result);
}

# 使用LinkedHashSet去重

private static void removeDuplicate(List<String> list) {
    LinkedHashSet<String> set = new LinkedHashSet<String>(list.size());
    set.addAll(list);
    list.clear();
    list.addAll(set);
}

# 使用HashSet去重

private static void removeDuplicate(List<String> list) {
    HashSet<String> set = new HashSet<String>(list.size());
    List<String> result = new ArrayList<String>(list.size());
    for (String str : list) {
        if (set.add(str)) {
            result.add(str);
        }
    }
    list.clear();
    list.addAll(result);
}

# 被序列化的类中的属性对象也需要支持序列化

findbugs会检测到这种场景，并提示异常 This Serializable class defines a non-primitive instance field which is neither transient, Serializable, or java.lang.Object, and does not appear to implement the Externalizable interface or the readObject() and writeObject() methods. Objects of this class will not be deserialized correctly if a non-Serializable object is stored in this field.

在支持序列化的类中，对其所有属性都需要同步支持序列化。

public class Child implements Serializable {
    private Father father;
}

// 如果Father不实现Serializable，则Child序列化会报错
public class Father {
    private String name;
}

# 父类属性和子类属性之间的关系

# 同名属性无法被覆盖

父类属性和子类同名属性会同时存在，子类可以使用super来引用父类的同名属性，如果不存在同名属性（子类无此属性），子类也可省略super直接使用父类属性。

# 实例方法引用当前类的属性

父类方法引用父类属性，不会引用子类同名属性，原因是父类方法无法感知到子类属性的存在。

# 父类方法如何使用子类属性？

  • 通过子类重写的方法获取子类属性值（需要维护两个同名属性）
  • 在子类构造方法中赋值给父类属性（使用super调用父类属性或者方法赋值）
  • 使用静态方法赋值（类似于上一种方式）
